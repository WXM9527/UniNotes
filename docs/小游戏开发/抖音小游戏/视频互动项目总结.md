## 视频互动项目总结



**关于代码架构**
出发点是 尽可能的保持和 Unity端相似的代码逻辑和结构
比如 事件的收发，数据和配置的读写，以及各个模块的划分和通讯
整体的逻辑编写尽可能只出现在语言差异上。
其次还有一个明显差异是 
HTML的UI处理是在HTML文件中填充对应的数据对象，只要更改这个数据对象，UI就会发生相应的变化。
并且不存在动态创建未声明的UI的方式。
所以 unity中一边计算数据然后动态生成UI的逻辑也需要相应的调整。



**关于视频组件**

​	首先说说 抖音视频组件 和 unity插件 api 的区别
	- 抖音的所有api都是**异步**的，无论是播放、暂停等基础操作
	- 没有直接的api可以获取当前**视频组件状态**，需要根据回调自行记录
	- 如果在视频回调（比如加载元数据成功，发送暂停命令）中直接做发送命令，这些命令很大部分不会被响应
	- 组件加载好元数据不代表可以播放，还是会出现转圈，前期就是为了解决转圈，才引入的多播放器组件。
	- 抖音视频组件 不设置 autoPlay，甚至不会加载元素据（组件预加载）
	- 设置 autoPlay 让他加载的同时 结束后会直接播放，导致如果使用多个播放器，还要额外加处理
	- 官方预加载api不支持m3u8（hls）格式，编辑器也对m3u8（hls）的支持有限
	- 多视频组件切换，没有故意设置延迟，只是因为没有做预加载
	- 某些mp4片段甚至部分机型无法播放，重新转码后才能正常播放。



1. **单视频组件**
   目前有个需求是，循环视频：第一次完整播放整个片段，后续循环在设定的时间内进行。

   但是遇到的问题是， 如果在播放结束的时候，直接进行seek操作，可能还是会从头开始播放，而忽略了我发送的操作指令。

   - 首先，这个命令不一定会被执行

   - 其次seek到0也是不被允许的，seek到0不会收到seek回调

   - 所以 seek 到 0 反而不需要额外的处理，让视频组件自己进行loop即可 

   - 所以目前是会在需要循环的时候，切换了一套处理逻辑，只会处理2个事件。

     ```js
         onVideoSeekEvent = (eventType) => {
             if (eventType == videoConst.VideoEventType.OnSeekFinished) {
                 this.videoComp.setCallBack(this, this.onVideoEvent)
             }
             else if (eventType == videoConst.VideoEventType.OnPlayStart) {
                 // 重新seek
                 this.timeoutList.startTimeout(() => {
                     this.videoComp.seekStart()
                 }, 1)
             }
         }
     ```

   - 单视频组件简单在于，只需要调整链接和使用seek操作。
